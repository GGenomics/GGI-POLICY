## Purpose

Entra ID and Exchange Online expose five overlapping group-shaped objects —
Security Group, Mail-enabled Security Group, Distribution Group, Microsoft
365 Group, and Shared Mailbox. Each has different access-control, email,
collaboration, and licensing implications. Choosing the wrong type creates
governance and security debt that is expensive to unwind: M365 Groups
created for what should have been a distribution list leak SharePoint sites
and Teams channels into the directory; security groups used for email
distribution accumulate orphaned permissions; shared mailboxes named after
people get re-purposed and forgotten.

This policy makes the type-selection decision deterministic, names every
group consistently so its purpose and scope are legible at a glance, and
binds ownership and privilege-management requirements to the group's
lifecycle. It is the foundation that downstream IAM policies (RBAC role
assignment, JML provisioning, access reviews) build on.

## Scope

In scope:

- All Entra ID Security Groups, including those synced from on-prem Active
  Directory and those created cloud-only.
- All Entra ID Mail-enabled Security Groups.
- All Exchange Online Distribution Groups (Distribution Lists), including
  hidden/internal-only.
- All Microsoft 365 Groups, including those created implicitly by Teams,
  SharePoint site provisioning, Yammer/Viva Engage, and Planner.
- All Exchange Online Shared Mailboxes.

Out of scope:

- Dynamic groups whose membership is managed by Entra ID rules (covered by
  a separate forthcoming policy on dynamic membership).
- On-prem-only AD groups that are *not* synced to Entra ID — those follow
  legacy AD conventions until decommissioned.
- Personal contacts, mail contacts, and shared calendars that are not
  themselves groups or mailboxes.
- Microsoft 365 Group display names auto-generated by SharePoint Online
  for non-collaboration sites (e.g., communication sites that elect not to
  back themselves with an M365 Group).

## Policy Statements

**R1.** Group type is chosen by the criteria in the decision table in the
sidecar (`policy.rules.yaml`, R1). The five permitted types are
`security_group`, `mail_enabled_security_group`, `distribution_group`,
`m365_group`, and `shared_mailbox`. No other group-shaped object is
created without an exception.

**R2.** Group display names and aliases conform to the type-specific
slug pattern:

| Type                          | Pattern                                                | Example                            |
| ----------------------------- | ------------------------------------------------------ | ---------------------------------- |
| Security Group                | `sg-<scope>-<env>-<purpose>`                           | `sg-az-prod-finance-readers`       |
| Mail-enabled Security Group   | `msg-<purpose>`                                        | `msg-secure-billing`               |
| Distribution Group            | `dl-<purpose>`                                         | `dl-all-engineering`               |
| Microsoft 365 Group           | `m365-<purpose>`                                       | `m365-platform-team`               |
| Shared Mailbox                | `sm-<purpose>`                                         | `sm-support`                       |

`<scope>` ∈ `{az, m365, app, file, infra}` (what the group grants access
to). `<env>` ∈ `{prod, nonprod, all}`. `<purpose>` is kebab-case ASCII
(`[a-z0-9-]+`). The full regex is in the sidecar (R2).

**R3.** Group names do not contain banned terms: `temp`, `test`, `tmp`,
`foo`, `bar`, `asdf`, `new`, `copy`, `copy-of`, `untitled`, `scratch`,
`todo`, `xxx`. Personal first or last names alone (`shannon-group`,
`smiths-group`) are also forbidden — every group describes its function,
not its current owner.

**R4.** Every group has at least two assigned owners who are
licensed user accounts (not service principals, not service accounts, not
shared mailboxes). One owner is designated primary, one is designated
backup. A group with fewer than two licensed-user owners is non-compliant
and routed to the IT Director within 30 days for remediation; failing
that, the group is archived per R7.

**R5.** Groups that grant any privileged role — including Entra ID
Global Administrator, Privileged Role Administrator, Conditional Access
Administrator, Security Administrator, User Administrator, or any Azure
RBAC `Owner`/`Contributor`/`User Access Administrator` role at
subscription or management-group scope — are managed via Privileged
Identity Management (PIM). Eligible-only assignments; no permanent
assignments. Activation requires MFA and approval per the role's PIM
policy. Direct (non-PIM) assignment to such a group is forbidden.

**R6.** Every group declares its lifecycle classification at creation:
`production`, `project`, or `temporary`. `temporary` groups carry a
documented expiry date in the description field (`expires: YYYY-MM-DD`)
and are reviewed by their owner monthly. `project` groups are reviewed
at project close-out. `production` groups are reviewed annually.

**R7.** Groups with no members for 90 days, or with no licensed-user
owner for 90 days, are archived: removed from any role assignments,
renamed with prefix `zz-archived-<original-name>`, and retained in a
read-only state for an additional 90 days before deletion. The IT
Director is notified before archival; group owners (if any) are
notified at archival.

## Rationale

**Why deterministic type selection (R1).** The active 2026 group-cleanup
work surfaced ~40% of M365 Groups created for use cases that should have
been Distribution Groups or Security Groups. The cost of correction is
high (Teams/SharePoint dependencies, re-permissioning, mailbox migration)
and almost entirely preventable by writing down the decision criteria
once. The decision table in R1 collapses the five-way choice into four
yes/no questions a non-IAM-specialist can answer.

**Why structured slugs (R2).** Group-shaped objects mix freely in the
GAL, the Entra portal, and PowerShell output. A glance-legible prefix
(`sg-`, `dl-`, `msg-`, `m365-`, `sm-`) tells the reader the type without
inspecting the object. Security-group facets (`scope`, `env`) further
disambiguate the most common type — operators routinely answer "is this
prod or nonprod, and what does it grant?" by reading the name. The
constraint also makes reports diff-able: searches for `sg-az-prod-*`
return everything that grants production Azure RBAC.

**Why two owners (R4).** Single-owner groups orphan when the owner
leaves; we have observed groups grant access to terminated employees'
former teams for 12+ months because no one knew they owned them. Two
licensed-user owners ensures continuity through routine departures and
gives a deterministic escalation contact for every group.

**Why PIM for privileged groups (R5).** Standing privileged-role
assignments are the principal compromise pathway for cloud tenants
(per Microsoft Entra recommendations and CIS 6.7). PIM converts those
assignments to time-bounded, MFA-gated activations, leaving no resident
high-privilege session at rest.

**Framework alignment.** R1–R3 map to NIST CSF PR.AA-01 (managed
identities) and PR.AA-03 (authentication of users and devices), CIS 5.1
(account inventory) and 5.4 (single primary account). R4 maps to CSF
GV.RR-04 (responsibilities and authorities), CIS 5.5 (admin acct
inventory), HIPAA 164.308(a)(3)(i) (workforce security), SOC 2 CC6.2
(user provisioning). R5 maps to CIS 6.7/6.8 (centralized access
control, privileged-account separation), HIPAA 164.312(a)(2)(i),
NIST 800-53 AC-5/AC-6, NIST 800-171 3.1.5 (least privilege). R7 maps
to CIS 5.3 (disable dormant accounts), AC-2(3) (disable accounts).

## Examples

**Compliant** — security group granting Azure RBAC reader on prod
finance subscription:

```
sg-az-prod-finance-readers
```

**Compliant** — distribution list for the Engineering org:

```
dl-all-engineering
```

**Compliant** — mail-enabled security group whose membership controls
both SharePoint access and a billing@ alias:

```
msg-secure-billing
```

**Compliant** — Microsoft 365 group backing a cross-functional
collaboration space with Teams + SharePoint:

```
m365-platform-team
```

**Compliant** — shared mailbox for the support inbox:

```
sm-support
```

**Non-compliant** (each fails the validator):

```
# Wrong type for the use case — needs ACL, so security_group, not m365_group
m365-az-prod-finance-readers          # violates R1 (decision table: ACL → security_group)

# Personal name as the group identity
sg-app-prod-shannons-tools            # violates R3 (forbidden term: personal name)

# Banned term in the name
dl-temp-marketing                     # violates R3 (forbidden term: 'temp')

# Distribution group used for resource ACL
dl-finance-prod-readers               # violates R1 (DL cannot grant ACL — should be sg-)

# M365 Group created for a use case that needs no files/chat
m365-quarterly-newsletter-recipients  # violates R1 (should be dl-quarterly-newsletter)

# Standing Global Admin assignment via group membership
sg-az-all-global-admins  (with permanent member assignments)
                                      # violates R5 (must be PIM-eligible)

# Single-owner group
sg-az-prod-finance-readers  (with one named owner)
                                      # violates R4 (requires two licensed-user owners)
```

## Implementation Guidance

**Authoring a new group:**

1. Walk the decision-table inputs (R1) on paper or in the issue request:
   - *Does this need files + chat (Teams/SharePoint)?*
   - *Does this need to receive email at a single address?*
   - *Does this need to grant access to a resource (file share, app,
     Azure resource)?*
   - *Is this a role-based mailbox (`support@`, `billing@`)?*
2. Map the answers to the type using R1's decision table.
3. Construct the name using R2's pattern. For security groups, identify
   the `scope` and `env` facets explicitly.
4. Lint the name locally before creating the group:

   ```bash
   uv run ggi-policy validate-name --rule POL-IAM-GROUP-NAMING.R2 --candidate sg-az-prod-finance-readers
   # CLI subcommand to be added in a follow-up PR;
   # in the meantime use the Python helper:
   python -c "from ggi_policy.enforce import evaluate; from ggi_policy.repo import repo_root; \
              r=evaluate(repo_root(), 'POL-IAM-GROUP-NAMING.R2', 'sg-az-prod-finance-readers'); \
              print(r.verdict, r.citation)"
   ```

5. Create the group in Entra (portal, Graph API, or Azure-AD PowerShell)
   with at least two licensed-user owners and the lifecycle classification
   (R6) recorded in the description.
6. If the group will grant any privileged role (R5), enroll it in PIM
   *before* assigning the role.

**Identifying privileged roles for R5.** The list is maintained in the
Entra Conditional Access policy CA-018 and is identical to Microsoft's
"Privileged roles in Microsoft Entra" reference list. A Conditional
Access policy blocks direct (non-PIM) role assignment to those roles;
violations of this policy will fail there before they fail here.

**Quarterly access review** (driven by R4 and R7):

- IT Director runs a Graph query for all groups missing two licensed-user
  owners and produces a remediation list.
- IT Director runs a Graph query for groups with zero members > 90 days
  or zero licensed-user owners > 90 days; archives per R7.
- Reports are filed against this policy ID and visible in the rendered
  site.

**Migrating an existing non-conformant group.** The advance-notice window
(2026-05-03 → 2026-06-15) exists specifically for this. The migration
path is in Revision History below. After 2026-06-15, the validator will
fail PRs that introduce new non-conformant groups; existing
non-conformant groups have until 2026-12-15 to be remediated or covered
by a written exception under `exceptions/`.

## Exceptions

See `exceptions/` for active exceptions referencing this policy. To
request an exception (e.g., a vendor-mandated group name that cannot
match R2, a legacy AD-synced group whose rename would break a
production integration), file a PR adding
`exceptions/EXC-{YYYY}-{NNN}-{slug}.md` with:

- `policy_ref: POL-IAM-GROUP-NAMING.R{n}` (the specific rule)
- A documented compensating control (e.g., monitored mailflow rule, an
  approved Conditional Access exclusion)
- A risk acceptance and an owner

Exception duration is capped per [POL-META-DOC-FRAMEWORK.R10] —
6 months for `required` rules, 18 months for `recommended` rules.
